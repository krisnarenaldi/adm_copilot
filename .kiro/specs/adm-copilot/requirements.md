# Requirements Document

## Introduction

ADM Copilot is an AI-powered travel audit assistant that automates the investigation of Agency Debit Memos (ADMs) — financial penalties issued by airlines to travel agencies. Auditors currently review lengthy airline Fare Rules PDFs manually to determine whether an ADM is valid or disputable, a process that is slow, error-prone, and risks missing dispute deadlines.

The system provides a web dashboard where authenticated users upload an ADM PDF, select the relevant airline, and receive an AI-generated audit verdict along with a ready-to-send dispute letter. It uses a Retrieval-Augmented Generation (RAG) architecture to cross-reference the ADM against stored airline Fare Rules, then outputs a structured verdict and formal dispute draft.

## Glossary

- **ADM (Agency Debit Memo)**: A financial penalty document issued by an airline to a travel agency for a claimed policy violation.
- **Fare Rules**: Airline policy documents (PDF or text) that define ticketing rules, penalties, booking conditions, and fare restrictions.
- **RAG (Retrieval-Augmented Generation)**: An AI architecture that retrieves relevant document chunks from a vector database and uses them as context for an LLM to generate grounded responses.
- **Auditor**: A travel agency revenue auditor or BPO travel operations agent who uses the system to investigate ADMs.
- **System**: The ADM Copilot application, comprising the Next.js frontend and FastAPI backend.
- **Frontend**: The Next.js web application hosted on Vercel.
- **Backend**: The FastAPI application hosted on Hugging Face Spaces via Docker.
- **Vector_Store**: The ChromaDB persistent vector database containerized inside the Hugging Face Spaces Docker image.
- **LLM**: The Gemma language model accessed via LangChain/Google Python SDK used for audit analysis and dispute draft generation.
- **Auth_Service**: The authentication layer backed by a Supabase PostgreSQL `users` table (agent_travel_name, email, and password).
- **Rate_Limiter**: The backend component that tracks and enforces per-user upload quotas using the Supabase PostgreSQL `user_uploads` table.
- **Ingestion_Pipeline**: The offline process that parses Fare Rules PDFs, chunks the text, and stores embeddings in the Vector_Store.
- **Audit_Pipeline**: The end-to-end backend workflow triggered by an ADM upload: JWT verification → rate-limit check → text extraction → vector retrieval → LLM analysis.
- **Verdict**: The binary audit outcome — either `VALID DISPUTE FOUND` or `VALID ADM / NO DISPUTE`.
- **Dispute_Draft**: A formal business-English email template generated by the LLM arguing the case for disputing the ADM.
- **airline_code**: A short identifier for an airline (e.g., `GA` for Garuda Indonesia) used as a metadata filter in the Vector_Store.
- **JWT**: A JSON Web Token issued upon successful login, used to authenticate requests to the Backend.

## Requirements

### Requirement 1: User Authentication

**User Story:** As an Auditor, I want to log in with my email and password, so that only authorized users can access the audit tools.

#### Acceptance Criteria

1. THE Auth_Service SHALL require an email address conforming to RFC 5322 format, a password of at least 8 characters, and a non-empty agent_travel_name to authenticate a user.
2. WHEN a user submits valid credentials, THE Auth_Service SHALL issue a JWT with a 24-hour expiry that the Frontend stores in memory or secure storage for subsequent requests.
3. WHEN a user submits invalid credentials, THE Auth_Service SHALL return an HTTP 401 response with a generic error message that does not indicate which field (email or password) is incorrect.
4. WHILE a user is not authenticated, THE Frontend SHALL restrict access to all audit features and redirect the user to the login page.
5. WHEN a JWT expires, THE Frontend SHALL redirect the user to the login page and clear the stored token.
6. WHEN a user submits invalid credentials 5 or more consecutive times for the same email address, THE Auth_Service SHALL lock that account for 15 minutes and return an HTTP 429 response indicating the lockout duration.
7. IF the Auth_Service is unavailable when a login request is submitted, THEN THE Frontend SHALL display an error message indicating that the service is temporarily unavailable and SHALL NOT redirect the user away from the login page.

### Requirement 2: Rate Limiting

**User Story:** As a system operator, I want to limit each user to a configurable number of ADM analyses per 24-hour period, so that API costs are controlled and the service remains available to all users.

#### Acceptance Criteria

1. THE Rate_Limiter SHALL read the maximum upload limit from an environment variable named `MAX_UPLOADS_PER_DAY`, which SHALL accept integer values between 1 and 100 inclusive, and SHALL default to 5 if the variable is absent or invalid.
2. IF a user's upload count within the current 24-hour rolling window has reached the value of `MAX_UPLOADS_PER_DAY`, THEN THE Rate_Limiter SHALL reject the upload request before the Audit_Pipeline executes.
3. WHEN a user submits an ADM upload request, THE Backend SHALL query the Rate_Limiter and verify the user's upload count before executing the Audit_Pipeline.
4. THE Rate_Limiter SHALL record each successfully processed upload in the Supabase PostgreSQL `user_uploads` table with the fields `user_email`, `upload_count`, and `last_upload_date`; rejected requests (HTTP 429) SHALL NOT increment the counter.
5. WHEN a user's upload count reaches the configured limit within the 24-hour rolling window, THE Backend SHALL return an HTTP 429 response containing the timestamp at which the quota resets, calculated as 24 hours after the user's earliest upload in the current window.
6. WHEN THE Backend returns an HTTP 429 response, THE Frontend SHALL display an inline message on the current page — without navigating away or blocking other page elements — informing the user of the quota limit and the exact reset timestamp from the response.
7. THE Rate_Limiter SHALL enforce the upload quota exclusively on the Backend so that quota enforcement cannot be bypassed by modifying Frontend requests.

### Requirement 3: Fare Rules Knowledge Base Ingestion

**User Story:** As a system operator, I want airline Fare Rules to be parsed, chunked, and stored in the vector database with airline metadata, so that the Audit_Pipeline can retrieve only the relevant policy sections for a given airline.

#### Acceptance Criteria

1. THE Ingestion_Pipeline SHALL parse Fare Rules documents in PDF and plain text formats using PyMuPDF; IF a document cannot be parsed (e.g., corrupt file, unsupported encoding), THEN THE Ingestion_Pipeline SHALL log the error and skip that document without halting the overall ingestion run.
2. THE Ingestion_Pipeline SHALL split extracted text into chunks of exactly 1000 characters with a 100-character overlap between consecutive chunks, cutting at the character boundary even if it falls mid-word.
3. THE Ingestion_Pipeline SHALL store each chunk in the Vector_Store with a metadata tag containing the `airline_code` field (e.g., `{"airline_code": "GA"}`); IF a document is submitted to the Ingestion_Pipeline without a valid `airline_code`, THEN THE Ingestion_Pipeline SHALL reject that document and log an error.
4. WHEN the Vector_Store is queried with a valid `airline_code`, THE Vector_Store SHALL return only chunks whose metadata `airline_code` field matches the queried value, up to a maximum of 20 chunks ranked by relevance score.
5. THE Vector_Store SHALL be initialized and queryable when the Backend starts using a pre-built static directory, but SHALL support persistent runtime updates.
6. THE Backend SHALL provide a secure API endpoint (`POST /fare-rules`) that allows dynamic uploading of Fare Rules PDF or plain text files; this endpoint SHALL require a valid JWT in the request header, a valid `airline_code`, and a single document file.
7. WHEN a valid dynamic ingestion request is processed, THE Backend SHALL parse the document using PyMuPDF, split it according to the standard chunking algorithm, and append the new chunks to the Vector_Store with the corresponding `airline_code` metadata.
8. IF a dynamic ingestion request lacks a valid JWT, THEN THE Backend SHALL return an HTTP 401 response; IF it lacks required parameters (such as the file or `airline_code`), THEN THE Backend SHALL return an HTTP 422 response.

### Requirement 4: ADM Input Selection

**User Story:** As an Auditor, I want to select an airline and upload an ADM PDF in a single step, so that I can initiate an audit quickly and without confusion.

#### Acceptance Criteria

1. THE Frontend SHALL provide an airline selector in combobox format that accepts both airline codes (e.g., `GA`) and full airline names (e.g., `Garuda Indonesia`) as search inputs and filters the dropdown list to matching entries as the user types.
2. THE Frontend SHALL provide a file drop zone that accepts exactly one PDF file per submission, with a maximum file size of 10 MB; IF a user drops multiple files simultaneously, THEN THE Frontend SHALL accept only the first file and discard the rest.
3. IF a user attempts to submit without selecting an airline, THEN THE Frontend SHALL display a validation error message directly below the airline selector and prevent submission.
4. IF a user attempts to submit without uploading a PDF file, THEN THE Frontend SHALL display a validation error message directly below the file drop zone and prevent submission.
5. IF a user attempts to upload a file whose MIME type is not `application/pdf` or whose file extension is not `.pdf`, THEN THE Frontend SHALL display a validation error message directly below the file drop zone and prevent submission.
6. IF a user drops a new PDF file onto the drop zone when a file is already selected, THEN THE Frontend SHALL replace the previously selected file with the new file.
7. THE Frontend SHALL require explicit user action (clicking the submit button) to initiate submission, even when a valid PDF file and airline selection are already present.
8. WHEN a user submits the form, THE Frontend SHALL send the PDF file and the selected `airline_code` to the Backend along with the user's JWT in the `Authorization: Bearer` request header.
9. THE Frontend SHALL fetch the list of supported airlines dynamically from the Backend upon mounting the dashboard; each airline entry SHALL contain both a unique airline code and a full name retrieved from the database.


---

### Requirement 5: Backend Audit Processing

**User Story:** As an Auditor, I want the system to automatically extract text from my ADM, retrieve the relevant Fare Rules, and query the AI, so that I receive an accurate audit result without manual research.

#### Acceptance Criteria

1. WHEN the Backend receives an ADM upload request, THE Backend SHALL verify the JWT before executing any further processing steps.
2. WHEN the JWT is invalid or absent, THE Backend SHALL return an HTTP 401 response and halt processing.
3. WHEN the JWT is valid, THE Backend SHALL check the Rate_Limiter; IF the rate limit has not been reached, THEN THE Backend SHALL proceed to extract raw text from the uploaded ADM PDF using PyMuPDF.
4. THE Backend SHALL complete ADM text extraction in under 500ms, measured from the moment the PDF bytes are available in memory to the moment raw text is returned by PyMuPDF.
5. WHEN text extraction is complete, THE Backend SHALL query the Vector_Store using the provided `airline_code` as a metadata filter to retrieve relevant Fare Rules chunks.
6. WHEN relevant Fare Rules chunks are retrieved, THE Backend SHALL compile the ADM text and the retrieved chunks into a structured prompt and dispatch it to the LLM.
7. WHEN the LLM returns a well-formed response, THE Backend SHALL parse the response into three components: a Verdict, an analysis explanation, and a Dispute_Draft.
8. IF text extraction from the ADM PDF fails, THEN THE Backend SHALL return an HTTP 422 response with a descriptive message indicating the extraction failure.
9. IF the Vector_Store returns no chunks for the specified `airline_code`, THEN THE Backend SHALL return an HTTP 422 response indicating that no Fare Rules are available for the selected airline.
10. IF the LLM call fails or times out, THEN THE Backend SHALL return an HTTP 502 response with a message indicating that the AI service is temporarily unavailable.
11. IF the LLM returns a response that cannot be parsed into the three required components, THEN THE Backend SHALL return an HTTP 422 response with a message indicating that the AI response was malformed.

---

### Requirement 6: Interactive Results Display

**User Story:** As an Auditor, I want to see a clear verdict, a detailed policy analysis, and a ready-to-send dispute letter, so that I can act on the audit result immediately.

#### Acceptance Criteria

1. WHEN the Backend returns a successful audit response, THE Frontend SHALL display exactly one Verdict badge at a time, removing any previously displayed badge before rendering the new one to prevent conflicting styling states.
2. WHEN the Verdict is `VALID DISPUTE FOUND`, THE Frontend SHALL render the Verdict badge with a green background and white text; IF the green background style cannot be applied due to a CSS failure, THEN THE Frontend SHALL hide the badge element entirely rather than displaying it unstyled.
3. WHEN the Verdict is `VALID ADM / NO DISPUTE`, THE Frontend SHALL render the Verdict badge with a red background and white text.
4. THE Frontend SHALL display the analysis explanation as a rendered markdown block; the rendered output SHALL include at least one reference to a specific policy clause, date, booking class, or penalty amount extracted from the Fare Rules chunks.
5. THE Frontend SHALL display the Dispute_Draft in a dedicated read-only text area pre-populated with a formal business-English email template generated by the LLM.
6. THE Frontend SHALL render a "Copy to Clipboard" button positioned immediately adjacent to the Dispute_Draft text area, visible without scrolling when the text area is in view.
7. WHEN a user clicks the "Copy to Clipboard" button, THE Frontend SHALL copy the complete, untruncated Dispute_Draft text to the user's system clipboard and display a visible confirmation indicator (e.g., a checkmark or "Copied!" label) for at least 2 seconds before reverting to the default button state.

---

### Requirement 7: Step-by-Step Processing Feedback

**User Story:** As an Auditor, I want to see real-time progress indicators while the system processes my ADM, so that I understand the system is working and know which step is currently executing.

#### Acceptance Criteria

1. WHEN an ADM upload is submitted, THE Frontend SHALL display a step-by-step processing tracker that advances sequentially through stages as the upload request lifecycle progresses, driven by the Frontend's own request state rather than server-sent events.
2. THE Frontend SHALL display exactly three processing stages in the following fixed order: "Extracting text...", "Matching rules...", and "Querying AI..."; each stage SHALL have exactly one of three mutually exclusive visual states — pending, active, or completed — with exactly one stage active at any given time during processing.
3. WHILE the Audit_Pipeline is executing, THE Frontend SHALL disable the submit button, the file drop zone, and the airline selector to prevent duplicate submissions.
4. IF an error occurs during Audit_Pipeline execution, THEN THE Frontend SHALL immediately re-enable the submit button, the file drop zone, and the airline selector to allow immediate correction of input choices, while displaying the error message with a dedicated dismiss control.
5. WHEN the Audit_Pipeline completes successfully, THE Frontend SHALL hide the processing tracker and display the audit results.
6. WHEN the user clicks the dismiss control on an error message, THE Frontend SHALL hide the processing tracker and the error message.


---

### Requirement 8: Responsive Desktop Layout

**User Story:** As an Auditor working on a desktop workstation, I want the interface to be optimized for large screens, so that I can view the full audit workflow and results without excessive scrolling or layout issues.

#### Acceptance Criteria

1. THE Frontend SHALL render the audit dashboard using a multi-column layout on viewport widths of 1280px and above, with the input selection panel and the results panel displayed side by side in separate columns.
2. WHEN the viewport width is 1280px or above, THE Frontend SHALL display the input selection panel, processing tracker, and results panel within the visible viewport height without requiring vertical scrolling to see all three panels simultaneously.
3. WHEN the viewport width is below 1280px, THE Frontend SHALL switch to a single-column stacked layout in which the input selection panel appears above the results panel, with no horizontal overflow or clipped content.
4. THE Frontend SHALL remain fully operable (all interactive controls reachable and usable) at viewport widths as narrow as 375px.
